AOP의 이해와 활용

🗓 2019-03-17
⏱ 예상 소요시간 - [12 분]
aop, aspect, proxy, spring

Intro

스프링의 AOP 프록싱 메커니즘에 대해 정리한다. 스프링 AOP에 대한 자세한 글은 링크를 참고하길 바란다. 거의 번역 위주의 글에 약간의 코드를 끼얹은 글이므로 여유롭다면 이 글보다 링크를 직접 보는 것을 추천한다.

다음과 같은 분들에게 이 글을 읽기를 권장한다.

  • AOP가 Aspect Oriented Programming이라는 사실을 알고있음
  • 스프링에서 간단하게나마 사용 정도는 해봤음
  • 위의 두 가지를 만족하고 공식 문서 읽을 시간이 없음

1. AOP 기초 용어

AOP의 핵심 컨셉과 용어를 먼저 정리한다. 스프링에 국한된 것이 아니라 AOP의 용어이다.

  • Aspect: 여러 클래스에 걸친 횡단 관심사의 모듈이다. Transaction Manager(트랜잭션 매니저)가 횡단 관심사의 대표적인 좋은 예라 볼 수 있다. 스프링 AOP에선 일반적인 클래스(스키마 기반 접근법)나 @Aspect 어노테이션이 붙은 클래스로 구현한다. OOP에서의 모듈이 객체인 것처럼 AOP에서의 모듈은 애스펙트이다.

  • Join point: 메소드의 실행이나 예외 처리와 같은 프로그램의 실행 시점을 의미한다. 쉽게 말해서 바로 뒤에 언급될 Advice(어드바이스)가 적용될 위치를 의미한다. 스프링 AOP에서 조인포인트는 항상 메소드 실행을 의미한다.

  • Advice: 특정한 JoinPoint에서 Aspect가 할 행동을 의미한다. 종류에는 around, before, after가 있다. 정리하자면 애스펙트의 행동과 행동을 할 시점을 정의한다고 보면 된다.

  • Pointcut: 매칭될 조인 포인트를 지정한다. 어드바이스는 포인트컷 표현식과 연관되어 있으며 포인트컷에 부합되는 모든 조인 포인트에서 실행된다. 스프링은 디폴트로 AspectJ의 포인트컷 표현식을 사용한다. 쉽게 말해 적용 대상을 의미한다. 다음을 떠올리면 쉽다. ("execution(* com.xyz.someapp..service.*.*(..))"))

  • Introduction: 타겟 클래스에 추가적인 메소드나 필드를 정의할 수 있게 해준다. 코드 수정이 발생하지 않는다. AspectJ 커뮤니티에서 인트로덕션은 inter-type 선언이라고도 부른다.

  • Target object: 하나 이상의 애스펙트에 의해 어드바이스 되는 객체를 의미한다. Advised Object라고도 부른다. 스프링 AOP에선 런타임 프록시를 사용해 구현되며 타겟 객체는 항상 프록시된 객체이다. 이게 궁금하면 프록시된 대상 객체에 getClass()를 때려보면 알 수 있다.
    class xyz.hyeyoom.logtutorial.TestController$$EnhancerBySpringCGLIB$$f79791f7

  • AOP proxy: AOP 프레임워크가 생성한 객체를 의미한다. 스프링 프레임워크에서 AOP proxy는 JDK dynamic proxy나 CGLIB proxy이다. 앞선 문단에서 클래스를 출력했을 때 SpringCGLIB이라는 것을 볼 수 있는데 스프링 부트는 cglib을 사용하기 때문이다. 사족이지만 이것 때문에 final이 붙은 경우나 등록된 bean이 아닌 디폴트 생성자가 없는 친구들의 경우에 오동작 아닌 오동작을 맛본 경험이 있을 것이다.

  • Weaving: 타겟 객체와 애스펙트를 연결하여 어드바이스 객체(Advised object)를 생성하는 것을 의미한다. 이는 컴파일 타임(예로 AspectJ 컴파일러 사용), 로드 타임, 런타임에 수행될 수 있다. 다른 자바 AOP 프레임워크와 마차낙지로 스프링 AOP는 런타임에 위빙을 수행한다.

2. Proxying Mechanism

AOP를 처음 접하고 사용해보면 이를 어떻게 구현했는지 상당히 궁금할 것이다. 하지만 예상 외로 상식적인 것을 사용해서 구현한다.
바로 프록시 패턴이다. 여기서는 프록시 패턴을 설명하지는 않고 사용을 해볼 것이다.

사용해보기 이전에 스프링의 프록시는 무엇으로 구현되었나 정리해보자.

프록시될 Target Object가 한 개 이상의 인터페이스의 구현체인 경우 JDK dynamic proxy가 사용된다. Target Object의 모든 인터페이스에 프록시가 붙는다. 인터페이스 구현체가 아니라면 CGLIB proxy가 사용된다.

CGLIB proxy에 대해 알아야 할 점은 다음와 같다.

  • final 메소드는 오버라이드 불가능하기 때문에 어드바이스 될 수 없다.
  • 스프링 3.2부터 spring-core에 기본적으로 포함되어 있다.
  • 스프링 4.0부터 CGLIB 프록시 인스턴스가 Objenesis를 통해 생성된 이후로는 프록시 객체의 생성자가 더 이상 두번 호출되지 않게 되었다.
    • 그러나 JVM이 생성자 바이패스를 허용하지 않는 경우에는 두 번 호출될 수 있다.

CGLIB을 강제하고 싶은 경우에는 spring.aop.proxy-target-class=true를 설정해주면 된다.

2.2 Spring AOP

쉽다. 쓰기만 하면 된다. 다만 쉽게 쓰기 위해서는 다음을 기억하자. 앞서 정리했던 용어들이므로 읽고 다시 돌아와도 좋다.

  • 무엇을 분리할 것인가? - Aspect
    • 여러 모듈을 관통하여 사용되는 것. 예를들어 트랜잭션, 로깅 등
  • 언제 하는가? - Advice
    • around, before, after, after returning, after throwing
    • 타겟의 실행 시점의 전, 후, 모두, 반환, 예외 등의 시점을 정함
    • 쉽게 말해서 분리된 관심사가 실행되는 시점
  • 어디서 하는가? - Pointcut
    • 분리된 관심사가 실행될 곳을 의미

스프링에선 오직 메소드의 조인포인트만 제공하기 때문에 JoinPoint를 별도로 언급하지 않았다. AOP에 대해 심도있게 공부하려면 AspectJ를 더 공부해볼 것을 추천한다.

뭐 어쨋든 본론으로 돌아와서 다음의 (일어날 수 없는) 시나리오를 읽고 위에서 말한 접근법으로 풀어보자.

  • API 서버의 응답 시간이 굉장히 느린 부분이 존재한다
  • 의심가는 18가지(임의의 숫자임. 의미부여 ㄴㄴ) 후보 모듈이 존재한다
  • 그런 이유로 모듈 실행 시간을 측정해야해
  • 필요하면 DB에 저장도 하고 싶음
  • 아.. DB 얘기가 나와서 말인데 쉽게 식별하기 위해 태그도 달아보고 싶음
  • 의심 후보군이 늘어날 수도 있으니 어노테이션으로 쉽게 쓰고싶다

그나저나 시나리오를 쓰고보니까 길다. 잡설은 그만두고 이제 질문을 해보자.

(1) 무엇을 분리할 것인가?

지정된 메소드 혹은 클래스의 실행 시간을 측정하는 모듈을 분리한다. 이름은 정해졌다. LoggingAspect이다.

(2) 언제 하는가?

수행 시간을 측정하려면 메소드의 시작과 끝을 알아야하므로 전과 후 모두 포함한다. 즉 Around이다.

(3) 어디서 하는가?

어노테이션이 붙은 모든 곳. 물론 의심했던 18가지 녀석에게 붙일 것이고 추가적으로 더 생길 수도 있다.
어노테이션의 이름도 정해졌다. LoggingPerformance이다.
